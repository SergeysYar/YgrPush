# Данные и база

## Основная база данных

Главный источник данных — `production.db`.

Система не должна ломать производственную базу, поэтому логика построена так, чтобы читать её в первую очередь как источник исторических и текущих данных, а собственные ML-артефакты хранить отдельно.

## Основные таблицы

### `measurements`

Главная производственная таблица.

Одна строка соответствует одному выполненному этапу или измерению варки.

Особенно важны поля:

- `id`
- `timestamp`
- `batchID`
- `loading_step_id`
- `loading_step_type`
- `component_1 ... component_7`
- `mass_1 ... mass_7`
- температурные поля
- pH-поля
- PE/frequency/duration поля

### `Batches`

Сведения о партии:

- `batch_id`
- `product_id`
- `batch_number`
- `production_date`
- `status`

### `Products`

Сведения о продукте:

- название
- категория
- база
- коды и технологические признаки

### `Components`

Справочник компонентов:

- `component_id`
- `name`
- `function_1..function_3`

Используется для:

- интерпретации;
- группировки компонентов;
- объяснения прогнозов.

### `Loading_Process`

Таблица порядка этапов.

Она важна, потому что порядок по `loading_step_id` сам по себе не гарантирует технологическую последовательность.

Система использует:

1. `Loading_Process.step_order`
2. затем `timestamp`
3. затем `id`

### `Testing_Protocols` и `testing_protocol_values`

Источник target-значений:

- `ph`
- `viscosity`
- `chlorides`

Числовые поля из `testing_protocol_values` считаются более надёжными, чем текстовые значения.

## Как формируются target-значения

Для каждой варки система пытается извлечь:

- итоговый `ph`
- итоговую `viscosity`
- итоговые `chlorides`

Поддерживается политика выбора протокола:

- `latest`
- `first`
- `all`

В текущем прикладном режиме по умолчанию используется `latest`.

## Очистка и нормализация

### Разбор чисел

Система умеет разбирать:

- `12.5`
- `12,5`
- строки с пробелами
- `None`

Некорректные значения не приводят к падению — они превращаются в `None` / `NaN` и отдельно отражаются в data quality.

### Проверки корректности

Например:

- `pH` вне диапазона `0..14` считается некорректным;
- отрицательная длительность считается ошибкой;
- пропуски в ключевых полях попадают в report.

## Component normalization

Компоненты разворачиваются из формата:

- `component_n`
- `mass_n`

в более пригодный аналитический вид.

Дополнительно система:

- подтягивает имя компонента;
- определяет группу компонента;
- считает агрегаты по компонентам.

## Batch dataset и snapshot dataset

### Batch dataset

Одна строка = одна варка.

Используется для финального анализа по завершённой партии.

### Snapshot dataset

Одна строка = один checkpoint одной варки.

Используется для промежуточного прогноза без утечки будущих этапов.

## Data quality layer

Система строит report по качеству данных и теперь умеет:

- считать summary;
- показывать sample issues;
- сохранять issues в `ml_storage.db`;
- отдавать stored issues через API.
