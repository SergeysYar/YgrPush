# Результаты и метрики

Этот документ усиливает проект как исследовательскую и презентационную работу. Его задача — показать, как в системе следует представлять результаты экспериментов, какие метрики считать ключевыми и как интерпретировать качество модели в промышленном контексте.

## Зачем нужен отдельный раздел результатов

Сильный научный проект отличается не только хорошей постановкой задачи, но и тем, что его результаты:

- представлены структурированно
- интерпретируются в контексте задачи
- сравниваются между собой по единым правилам
- отделяют offline-качество от реальной практической полезности

Именно поэтому результаты лучше оформлять отдельно, а не оставлять их только в логах обучения или в базе артефактов.

## Какие метрики стоит считать базовыми

Для регрессионной задачи раннего прогнозирования качества рекомендуется держать следующий минимальный набор:

- `MAE`
- `RMSE`
- `Median AE`
- `R²`

Почему именно они:

- `MAE` удобно интерпретируется в предметных единицах
- `RMSE` сильнее штрафует крупные ошибки
- `Median AE` устойчивее к отдельным выбросам
- `R²` даёт общее представление об объяснённой вариативности

## Почему одной метрики недостаточно

Для промышленного сценария важно не только “среднее качество”, но и характер ошибок:

- бывают ли редкие, но большие промахи
- насколько модель стабильна между партиями
- как меняется точность на ранних этапах процесса
- есть ли таргеты, по которым модель особенно ненадёжна

Поэтому хороший раздел результатов всегда должен сочетать несколько метрик и содержательную интерпретацию.

## Как представлять результаты по target

Для каждого целевого показателя полезно фиксировать:

- тип модели
- число партий в обучении
- число партий в валидации
- режим датасета: `batch-level` или `snapshot-level`
- набор доступных этапов
- `MAE`
- `RMSE`
- `Median AE`
- `R²`
- краткий комментарий по качеству

### Пример структуры таблицы

| Target | Model | Dataset mode | MAE | RMSE | Median AE | R² | Notes |
|---|---|---|---:|---:|---:|---:|---|
| pH | Ridge | snapshot | ... | ... | ... | ... | устойчиво на средних этапах |
| viscosity | BayesianRidge | snapshot | ... | ... | ... | ... | чувствительно к поздним измерениям |
| chlorides | PLS | batch | ... | ... | ... | ... | нужен анализ данных по выбросам |

## Как показывать результаты по горизонту прогноза

Одна из самых сильных частей этого проекта — возможность анализировать качество по мере накопления этапов варки.

Рекомендуется строить таблицы или графики вида:

- после шага 1
- после шага 2
- после шага 3
- после шага 4
- в финале

Это позволяет ответить на один из самых важных практических вопросов:

**На каком этапе прогноз уже достаточно хорош, чтобы приносить реальную производственную пользу?**

## Как интерпретировать хорошие результаты

Хороший результат в контексте этого проекта — это не обязательно максимальный `R²`. Гораздо важнее сочетание факторов:

- ошибка достаточно мала для технологической интерпретации
- качество стабильно на новых партиях
- прогноз полезен ещё до завершения варки
- модель не теряет доверия из-за полной непрозрачности

Поэтому в проекте особенно ценны модели, которые не просто “лучше по скору”, а дают устойчивый и объяснимый результат.

## Как интерпретировать слабые результаты

Если качество модели слабое, это не всегда означает, что идея проекта плоха. Возможные причины:

- мало исторических партий
- шумные или неполные измерения
- слабая связь между ранними этапами и итоговым target
- недостаточно информативные признаки
- высокая неоднородность продукта или режимов производства

Для научного проекта это тоже полезный результат: он помогает понять границы применимости подхода.

## Что особенно хорошо смотрится в презентации

Если нужно показать проект как сильную исследовательскую работу, лучший набор артефактов такой:

- сравнительная таблица моделей по каждому target
- анализ качества по шагам варки
- список наиболее влияющих признаков
- примеры типичных и нетипичных ошибок
- выводы о практической пользе раннего прогноза

## Рекомендуемый формат итогового блока результатов

Ниже удобная структура для отчёта, презентации или README:

1. Постановка задачи
2. Источник и характер данных
3. Схема построения датасета
4. Валидационный протокол
5. Сравнение baseline и candidate-моделей
6. Результаты по target
7. Результаты по этапам варки
8. Интерпретация ошибок
9. Практический вывод

## Практический вывод, который стоит уметь делать

Самый сильный финальный вывод для такого проекта звучит примерно так:

> Система не просто предсказывает итоговые показатели качества, а позволяет определить, на каком этапе технологического процесса становится возможен practically useful forecast. Это делает проект ценным не только как ML-эксперимент, но и как инструмент производственного decision support.
