# Примеры работы с API

Этот документ помогает быстро начать пользоваться API без чтения всего описания маршрутов. Здесь собраны типовые сценарии с практическими примерами.

## Когда использовать этот документ

Открывайте его, если вам нужно:

- быстро проверить, что API поднялся
- встроить систему в другой сервис
- показать интеграционный сценарий команде
- получить заготовки запросов для Postman, curl или внутреннего клиента

## Предпосылка

Считаем, что API уже запущен локально:

```bash
python -m app.cli run-api
```

Дальше в примерах используется базовый URL:

```text
http://localhost:8000
```

Если у вас другой порт или хост, просто подставьте свои значения.

## Сценарий 1. Проверка доступности API

Типовая цель — понять, что сервис вообще отвечает.

```bash
curl http://localhost:8000/
```

Что ожидается:

- HTTP-ответ без ошибки
- краткая сводка или служебная информация о сервисе

## Сценарий 2. Получение общей сводки

Полезно, когда нужно показать общее состояние системы или наполненность данных.

```bash
curl http://localhost:8000/summary
```

Обычно это используется для:

- обзорной страницы
- health-like проверки бизнес-уровня
- быстрой оценки готовности системы к использованию

## Сценарий 3. Просмотр партий

Если нужно выбрать партию для анализа:

```bash
curl http://localhost:8000/batches
```

Что это даёт:

- список доступных batch
- основу для выбора конкретной партии под прогноз

## Сценарий 4. Обучение модели через API

Если обучение разрешено через API, сценарий обычно выглядит так:

```bash
curl -X POST http://localhost:8000/train
```

В реальной эксплуатации такой вызов стоит использовать осторожно, потому что обучение:

- может занимать заметное время
- создаёт новые артефакты
- меняет операционное состояние ML-контура

## Сценарий 5. Прогноз по партии

Базовый пример:

```bash
curl -X POST http://localhost:8000/predict ^
  -H "Content-Type: application/json" ^
  -d "{\"batch_id\": 89}"
```

Что обычно ожидается в ответе:

- идентификатор партии
- прогнозы по таргетам
- информация о модели
- дополнительные служебные поля, если они предусмотрены реализацией

## Сценарий 6. Промежуточный прогноз по завершённым этапам

Если нужен прогноз только по состоянию на конкретный шаг:

```bash
curl -X POST http://localhost:8000/predict ^
  -H "Content-Type: application/json" ^
  -d "{\"batch_id\": 89, \"up_to_step\": 4}"
```

Это особенно полезно:

- в середине варки
- при сравнении ожиданий до и после нового этапа
- при построении сценариев оперативного управления процессом

## Сценарий 7. Просмотр моделей

Для диагностики доступных моделей:

```bash
curl http://localhost:8000/models
```

Обычно здесь хотят увидеть:

- список обученных моделей
- тип модели
- таргет
- статус активности
- метрики или служебные атрибуты

## Сценарий 8. Получение отчётов

Если API умеет отдавать отчёты:

```bash
curl http://localhost:8000/reports
```

Этот маршрут полезен для:

- dashboard
- внутренних отчётов
- анализа качества моделей и покрытия данных

## Рекомендации по интеграции

- Не считайте прогнозы окончательным качеством продукта
- Разделяйте промежуточные и финальные прогнозы в интерфейсе
- Логируйте идентификатор партии, модель и время прогноза
- Храните фактические значения отдельно, чтобы позже оценивать качество модели
