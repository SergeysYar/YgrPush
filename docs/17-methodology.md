# Методология проекта

Этот документ оформляет проект в более академичном стиле и показывает, как можно описывать его как методологически выстроенную исследовательскую работу.

## 1. Problem Statement

Цель проекта — построить систему раннего прогнозирования итогового качества партии шампуня на основе технологических измерений, доступных по мере прохождения этапов варки.

В формализованном виде задача состоит в следующем:

- есть партия `batch_id`
- есть последовательность измерений и состояний процесса
- есть итоговые лабораторные target-значения
- нужно предсказать эти target-значения до получения финального лабораторного результата

## 2. Data Sources

Проект использует несколько классов данных:

- производственные измерения
- batch-метаданные
- product-атрибуты
- лабораторные протоколы и целевые значения

Основной источник — `production.db`.

## 3. Data Processing

До обучения модели данные проходят несколько обязательных стадий:

- извлечение из базы
- очистка и нормализация значений
- согласование идентификаторов партий и связанных сущностей
- выделение target-показателей
- построение итоговых и промежуточных признаков

На этом этапе особенно важно сохранить корректную временную логику процесса.

## 4. Dataset Design

В проекте есть два центральных режима постановки:

- `batch-level`
- `snapshot-level`

### Batch-level

Используется, когда нужно описать партию по завершённому процессу или по максимально полному набору признаков.

### Snapshot-level

Используется, когда нужно имитировать реальный сценарий раннего прогноза. Для каждой партии может существовать несколько наблюдений, соответствующих разным этапам накопления информации.

Именно `snapshot-level` делает проект особенно интересным с исследовательской точки зрения.

## 5. Leakage Control

Один из ключевых методологических рисков в таких задачах — утечка будущей информации.

В проекте этот риск контролируется через принцип:

**Признаки в конкретной точке прогноза могут использовать только ту информацию, которая уже была доступна к этому моменту технологического процесса.**

Это делает эксперимент честным и приближённым к реальной эксплуатации.

## 6. Validation Protocol

Для корректной оценки модели используется `LeaveOneGroupOut` по `batch_id`.

Это означает:

- одна партия целиком попадает либо в train, либо в validation
- snapshots одной и той же варки не смешиваются между частями оценки
- метрика ближе к реальному поведению модели на новых партиях

## 7. Modeling Strategy

Проект специально строится от простого к сложному:

1. baseline
2. интерпретируемые линейные модели
3. расширенные нелинейные модели при необходимости

Такой подход хорош потому, что:

- снижает риск переусложнения
- даёт понятную отправную точку сравнения
- позволяет доказать ценность более сложной модели, если она действительно есть

## 8. Explainability

Для промышленного применения модель должна быть не только точной, но и объяснимой.

Поэтому в методологию проекта входят:

- анализ значимых признаков
- интерпретация коэффициентов для линейных моделей
- возможность расширения к SHAP/feature importance для более сложных моделей

## 9. Experiment Tracking

Научная и инженерная зрелость проекта усиливается тем, что обучение не воспринимается как одноразовый запуск. Вместо этого проект поддерживает:

- хранение training runs
- хранение версий моделей
- разделение champion и challenger
- сохранение метрик и артефактов

## 10. Evaluation Philosophy

Проект оценивает качество модели не по принципу “максимальный скор любой ценой”, а по более зрелой логике:

- честность постановки
- устойчивость на новых партиях
- полезность на ранних этапах процесса
- интерпретируемость результатов
- пригодность к внедрению

## 11. Threats to Validity

Как и у любой прикладной исследовательской работы, у проекта есть ограничения:

- ограниченный объём исторических данных
- возможные пропуски и шум в измерениях
- смещение по продуктам или типам партий
- различие между offline-метрикой и реальной полезностью в эксплуатации

Явное описание этих ограничений усиливает проект, а не ослабляет его.

## 12. Methodological Contribution

Методологический вклад проекта состоит в том, что он показывает, как можно объединить:

- industrial data engineering
- early prediction
- interpretable machine learning
- group-aware validation
- reproducible experimentation

в рамках одной практически полезной системы.
